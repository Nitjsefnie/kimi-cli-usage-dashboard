Design spec · kimi-dash

Model attribution: wire-first, per-record

Context kimi-dash labels every billing record with a pricing model and precomputes cost at parse time. Today that label comes only from hardcoded date cutoffs applied to a session's first event, and the wire's own model string is deliberately discarded. That scheme is wrong in four independently-verified ways. This spec replaces it with a per-record, wire-first assignment, keeping dates only where the wire genuinely cannot answer.

Decision
The wire string decides when present and unambiguous; dates decide only what the wire cannot express; both are evaluated per-record.
The kimi-code wire carries model directly on the usage.record — the exact record that becomes a billing row. A date is a proxy for a fact the transcript states outright, so the transcript wins.
This deliberately does not remove dates. Legacy-format transcripts carry no model string at all and are still being produced, so dates remain the only available signal for them.
01

Evidence

Every finding below was verified against the live corpus and database — none is inferred from the constants.

#FindingEvidence
1 The kimi-code wire carries the model on the usage.record itself — the exact record that becomes a billing row. L27 type=usage.record model='kimi-code/k3'
2 K3_CUTOFF_EPOCH is provably wrong. Real k3 records predate it by 20m41s and are billed at ~⅓ the true rate. earliest k3 record 2026-07-16 14:45:55.471Z; constant = 15:06:34Z
3 Model is not a property of a session. Session aed8326f switches model mid-session, 24s apart. L17 kimi-for-coding @14:45:34 → L27 k3 @14:45:55
4 Per-session labelling leaks across the boundary — records outlive their session's era. DB has kimi-k2-7-code records at 2026-07-16 15:10:14, after the cutoff
5 Legacy transcripts carry no model string at all — and are still being produced. Dates are unavoidable for them. R2 sessions/2aa458d2…, protocol_version 1.10, dated 2026-07-16 18:25
6 kimi-code/kimi-for-coding is ambiguous: it covers both k2.6 and k2.7-code — branding didn't change at that transition. only two distinct strings exist corpus-wide

Only two model strings exist across all 67 local sessions: kimi-code/kimi-for-coding and kimi-code/k3.

02

Before vs after

What the current scheme asserts, against what the corpus actually shows.

Current — date-only, per-session
  • Wire model strings are "deliberately ignored" as raw provider ids.
  • k3 before 15:06:34 is billed as k2.7-code (~⅓ rate).
  • k2.7-code after the K3 cutoff would be billed as k3 (~3× rate).
  • A mid-session model switch is unrepresentable.
  • Records are stamped from the session's first event.
Proposed — wire-first, per-record
  • kimi-code/k3 maps cleanly and unambiguously to kimi-k3.
  • k3 is k3 whenever the wire says so, whatever the date.
  • k2.7-code stays k2.7-code after the cutoff, by construction.
  • Each record resolves independently; switches are represented.
  • Dates apply only to model-less legacy records.
03

Resolution ladder

Applied to each billing record, in order. First match wins.

1
Wire model string says k3 → kimi-k3. Regardless of date.
2
Wire says kimi-for-coding → ambiguous era → the record's own timestamp vs MODEL_CUTOFF_EPOCH picks kimi-k2-6 (before) or kimi-k2-7-code (at/after). K3_CUTOFF_EPOCH is not consulted — a wire that says kimi-for-coding is not k3, whatever the date.
3
No / unrecognized model string (legacy format) → full date ladder on the record's own timestamp, exactly as today (both cutoffs).
4
No timestamp either → kimi-k2-7-code (unchanged fallback).

This fixes the reported bug by construction: k2.7-code used after the K3 cutoff hits rule 2 and stays kimi-k2-7-code. And k3 used before the cutoff hits rule 1 and is correctly kimi-k3.

04

Rejected alternatives

Both were considered and killed by specific evidence.

Delete the date cutoffs entirely
Rejected by finding 5. The legacy CLI is still live and emits no model string; k2.6 history is entirely legacy. Dates are the only available signal there.
Keep dates, just fix the wrong constant
Rejected: it cannot express finding 3 (mid-session switch) and can never distinguish post-cutoff k2.7 from k3 — the originally reported bug. A date is a proxy for a fact the wire states outright.

09

Out of scope

Explicitly deferred — named so the reader knows it was considered, not missed.

Backfilling a model string into legacy transcripts
Not possible — the data was never written.
Re-deriving MODEL_CUTOFF_EPOCH
Its to-the-second precision is fiction: k2.6 records stop 2026-06-10 22:42 and k2.7 starts 2026-06-12 15:23, so the constant sits in a 41-hour data gap. Any value in that window is equally defensible. Leave it.
Rate-table changes
Cost math is unchanged; only labels move.
